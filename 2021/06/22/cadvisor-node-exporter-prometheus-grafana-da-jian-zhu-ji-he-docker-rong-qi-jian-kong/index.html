cadvisor+node-exporter+prometheus+grafana搭建主机和docker容器监控

前言

目前缺乏针对主机和各个docker容器的监控，且缺乏全面的覆盖，需要构建全面覆盖从主机到docker容器的监控，甚至针对单个进程以及jvm的监控。

这样在选择监控平台的时候需要较广的覆盖面，监控指标到位，且得有良好的展示方式。基于以上需求，选择以prometheus为核心的监控体系，各个部分功能如下图：

利用各种exporter，收集对应的监控数据信息，全面覆盖从数据库到服务到主机层面的监控能力，甚至可以对单个特殊进程进行监控！

  1. mysqld-exporter

对于MySQL的监控，首先需要设定合适的用户，创建exporter上报的用户信息，如下：

$ mysql -u root -p
Enter password:

> GRANT REPLICATION CLIENT, PROCESS ON  *.*  to 'exporter'@'%' identified by '123456';
> GRANT SELECT ON performance_schema.* TO 'exporter'@'%';
> flush privileges;

如果执行MySQL命令时提示密码过于简单，可以使用以下的命令执行：

> GRANT REPLICATION CLIENT, PROCESS ON  *.*  to 'exporter'@'%' identified by 'ht@MySQL111';

> GRANT SELECT ON performance_schema.* TO 'exporter'@'%';

> flush privileges;

然后在MySQL所在的机器上安装exporter，如下：

$ docker pull prom/mysqld-exporter:v0.12.1

// 注意指定MySQL的连接信息
$ docker run -d  --restart=always  --name mysqld-exporter -p 9104:9104   -e DATA_SOURCE_NAME="exporter:ht@MySQL111@(10.0.93.100:3306)/"   prom/mysqld-exporter:v0.12.1

$ sudo firewall-cmd --add-port=9104/tcp --zone=public --permanent

$ sudo firewall-cmd --reload

以监控nacos进程为例子，进行配置，通过匹配nacos的进程信息，实现对nacos进程的监控。这里一共有四种进程监控方式：

  • “{{ .Comm }}” 包含原始可执行文件的basename，/proc//stat 中的换句话说，2nd 字段
  • “{{ .ExeBase }}” 包含可执行文件的basename
  • “{{ .ExeFull }}” 包含可执行文件的完全限定路径
  • “{{ .Username }}” 根据用户信息进行匹配
  • “{{ .Matches }}” 映射包含应用命令行所产生的所有匹配项

告警操作设置

使用Prometheus+Alertmanager进行告警的操作，分为两个部分：

  • Prometheus负责中配置警报规则，将警报发送到Alertmanager。
  • Alertmanager 负责管理这些警报，包括沉默，抑制，合并和发送通知。

Alertmanager 发送通知有多种方式，支持邮件和Webhook等报警方式，这里采用webhook的方式，通过对接企业微信机器人的形式，达到报警的目的！

主要分为以下四个方面：

  • prometheus报警规则配置
  • alertmanager部署及配置
  • 关联prometheus和alertmanager
  • 配置报警通知方式

prometheus报警规则配置

对prometheus配置文件进行调整，添加报警部分的配置，如下：

global:
  scrape_interval: 60s
  evaluation_interval: 30s

# 告警功能配置
alerting:
  alertmanagers:
    - static_configs:
      - targets:
        # - 192.168.166.203:9093

# 告警规则文件设置
rule_files:
  - "rules/*_rules.yml"

scrape_configs:
  - job_name: prometheus
    static_configs:
      - targets: ['localhost:9090']
        labels:
          instance: prometheus
......

这里临时注释了alertmanager的连接信息，待安装配置alertmanager时，放开即可。

添加prometheus报警规则，如下：


$ mkdir /home/centos/docker/prometheus/rules

$ vim host_rules.yml
groups:
- name: host-lost-connection-rule
  rules:
  - alert: ServerStatus
    expr: up == 0
    for: 2m
    labels:
      severity: page
      status: warning
    annotations:
      summary: "{{$labels.instance}}:服务器关闭"
      description: "{{$labels.instance}}:服务器关闭"
# 报警组组名称
- name: hostStatsAlert
  #报警组规则
  rules:
   #告警名称，需唯一
  - alert: hostCpuUsageAlert
    #promQL表达式
    expr: sum(avg without (cpu)(irate(node_cpu_seconds_total{mode!='idle'}[5m]))) by (instance) > 0.85
    #满足此表达式持续时间超过for规定的时间才会触发此报警
    for: 1m
    labels:
      #严重级别
      severity: page
    annotations:
      #发出的告警标题
      summary: "实例 {{ $labels.instance }} CPU 使用率过高"
      #发出的告警内容
      description: "实例{{ $labels.instance }} CPU 使用率超过 85% (当前值为: {{ $value }})"
  - alert: hostMemUsageAlert
    expr: (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)/node_memory_MemTotal_bytes > 0.85
    for: 1m
    labels:
      severity: page
    annotations:
      summary: "实例 {{ $labels.instance }} 内存使用率过高"
      description: "实例 {{ $labels.instance }} 内存使用率 85% (当前值为: {{ $value }})"

配置添加完成后，重启prometheus服务，如下：

$ docker restart prometheus

  group_by: ['alertname']
  group_wait: 30s
  group_interval: 60s

监控告警指标的确定

1. 主机层面的监控

满足以下条件进行告警：

  • CPU使用率，超过85%
  • 内存使用率，超过85%
  • 网络吞吐量，超过80MB/s
  • 网络传输错误，出现一次
  • 硬盘空间剩余，不足10%
  • 硬盘读写量不正常，超过50MB/s
  • 交换空间使用率，超过80%（限部分机器）

2. docker容器的告警

满足以下条件进行告警：

  • 容器被killed
  • CPU使用率，超过80%
  • 内存使用率，超过80%
  • 容器挂载的存储使用量，超过80%
  • 容器磁盘读写使用率，超过80%

3. MySQL的告警

满足以下条件进行告警：

  • MySQL宕机
  • 连接数占用过多，超过80%
  • 线程利用过高，超过60%
  • 慢查询，出现1次

*注意： 目前MySQL按照单机模式进行监控，并未以集群方式监控。

4. Redis的告警

满足以下条件进行告警：

  • Redis宕机
  • OOM，内存使用大于90%
  • 连接数过多，超过100
  • 连接数不足，小于5
  • 存在拒绝连接的情况

*注意： 目前Redis按照单机模式进行监控，并未以集群方式监控。

5. MongoDB的告警

满足以下条件进行告警：

  • 游标（cursor）开启数，大于10k
  • 游标访问超时的数量，大于100
  • 连接数过多，大于500
  • 虚拟内存使用量过高，超过1/3

6. 进程告警（临时未添加）

7. kafka告警

满足下面条件进行告警：

  • 处于同步状态中的topic数量，超过3个
  • 消费者组超过50个实例

总结

选择prometheus做监控，第一是容易建设，开箱即用，第二是生态组合优秀，监控面广。

经过建设后，在对主机管理层面，提升整体的管理水平，减少因硬件资源水平不足，例如硬盘空间不够，这样的问题带来对整个服务器集群的不良影响。
