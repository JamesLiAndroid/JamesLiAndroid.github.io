关于某项目k8s下网络问题排查记录

背景以及现象

最近在甲方环境中，安装了一套新的k8s服务，部署nginx进行了访问测试，测试通过。

在部署项目的后端微服务时，遇见了严重的问题，服务启动不了，首先是服务一直处在ContainerCreating状态，删除cni0网桥后，服务经常出现重启的情况，查看服务的具体日志可以看到，在k8s集群中部署的微服务，无法访问部署在节点上的nacos配置中心，出现了timeout的情况，导致服务的配置文件加载不到。

简单介绍下环境信息：

  • k8s版本：1.20.14，使用kubeadm的方式进行部署
  • coredns版本：1.7.0
  • 网络插件：flannel v0.15.1

最开始一切基础服务（kube-system命名空间下）都是运行正常的，至少看READY的状态都是1/1的，删除cni0网桥后，coredns出现了反复重启的情况。总结一下上面的问题，就是在pod中访问节点的ip地址，并不能正常访问！

解决思路

1. 安装busybox-curl增强工具，排查下网络问题


$ docker pull radial/busyboxplus:curl

// 将镜像导入到harbor镜像仓库
$ docker tag radial/busyboxplus:curl 192.168.223.141/test-conditions/busyboxplus:curl

$ docker push 192.168.223.141/test-conditions/busyboxplus:curl

// kubectl运行进入交互空间
$ kubectl run yourdockername --image=192.168.223.141/test-conditions/busyboxplus:curl  -i --tty --rm

// 访问nacos地址
[ root@yourdockername:/ ]$ curl -vvv -k -L http://192.168.223.130:18848/nacos
> GET /nacos HTTP/1.1
> User-Agent: curl/7.35.0
> Host: 192.168.223.130:18848
> Accept: */*
connection timeout

访问nacos地址超时，进一步验证了问题所在。剩下的需要看一下解析的内容，如下：


$ ping 192.168.223.130
// 不返回信息

$ nslookup 192.168.223.133
Server:    10.96.0.10
Address 1: 10.96.0.10 kube-dns.kube-system.svc.cluster.local

// 无法输出后半部分关于192.168.223.133的解析信息

根据上面的测试发现，目前访问时不通的。

2. 尝试删除cni0网桥，不能自动恢复

对每一个k8s节点，执行删除cni0网桥的操作如下：


$ ip addr | grep cni0
1719: cni0:  mtu 1450 qdisc noqueue state UP group default qlen 1000
    inet 10.240.4.1/24 brd 10.240.4.255 scope global cni0

// 删除cni0网桥
$ sudo ip link set cni0 down; sudo ip link delete cni0

删除cni0网桥后，发现coredns在反复重启，且一直报错，日志如下：

kubectl logs -f coredns-bccdc95cf-jt2pk -n kube-system

E0229 08:52:05.662547       1 reflector.go:134] github.com/coredns/coredns/plugin/kubernetes/controller.go:317: Failed to list *v1.Endpoints: Get https://10.96.0.1:443/api/v1/endpoints?limit=500&resourceVersion=0: dial tcp 10.96.0.1:443: connect: no route to host

根据上述问题，清理所有k8s节点机器的iptables规则，操作如下：

$ sudo systemctl stop kubelet
$ sudo systemctl stop docker
$ sudo iptables --flush
$ sudo iptables -tnat --flush
$ sudo systemctl start docker
$ sudo systemctl start kubelet

清理完成后问题依旧，而且cni0网桥并未恢复。

3. 删除flannel网络插件，再部署，问题依旧

flannel网络插件是通过yaml部署的，删除也很简单：


$ kubectl delete -f flannel.yaml

等所有flannel容器停机后，再重新部署如下：


$ kubectl apply -f flannel.yaml

重新部署后问题依旧，而且cni0网桥并未恢复。

4. 所有节点清理iptables后，开放iptables流入、流出以及允许转发，删除flannel网络插件，删除coredns容器，再重新部署flannel网络插件

在所有节点，清理iptables请参考第二部分的处理，开放iptables规则如下:


$ sudo iptables -P INPUT ACCEPT && sudo iptables -P FORWARD ACCEPT && sudo  iptables -P OUTPUT ACCEPT && sudo iptables -F

删除flannel网络插件和重新部署，参考第三部分的处理，删除coredns容器操作如下：


$ kubectl get pod -n kube-system | grep coredns
coredns-54d67798b7-7cl76                1/1     Running   0          30h
coredns-54d67798b7-llcqv                1/1     Running   0          30h

$ kubectl delete pod coredns-54d67798b7-7cl76 -n kube-system

$ kubectl delete pod coredns-54d67798b7-llcqv -n kube-system

重新部署flannel插件后，问题依旧，而且cni0网桥依旧未恢复。

5. 删除所有节点flannel.1网桥、cni0网桥，删除flanel网络插件，删除coredns重启，待coredns恢复后再重新部署flannel网络插件

对所有节点删除flannel.1网桥以及cni0网桥，操作如下：

$ sudo ip link set flannel.1 down; sudo ip link delete flannel.1 
$ sudo ip link set cni0 down; sudo ip link delete cni0

删除flannel网络插件如下：

$ kubectl delete -f flannel.yaml

删除两个coredns所在的pod，如下：


$ kubectl get pod -n kube-system | grep coredns
coredns-54d67798b7-7cl76                1/1     Running   0          30h
coredns-54d67798b7-llcqv                1/1     Running   0          30h

$ kubectl delete pod coredns-54d67798b7-7cl76 -n kube-system

$ kubectl delete pod coredns-54d67798b7-llcqv -n kube-system

删除后等待coredns恢复正常，READY状态为1/1。

最后重新部署flannel网络插件，如下：


$ kubectl apply -f flannel.yaml

等待flannel启动完毕，即可进行测试。

总结

其实在第五部分的操作中，也更建议清理下本机的iptables规则。但是当时正式环境中服务以docker方式启动，清理iptables规则需要停止docker服务，会影响目前系统运行，因此并未做这个事情。

这里需要说明的是，重置k8s集群固然很容易就解决这个问题，但是对你个人并不利，还是要多探索下如何操作能挽救，总要先尝试一下，实在不行了最后再去重置集群！
