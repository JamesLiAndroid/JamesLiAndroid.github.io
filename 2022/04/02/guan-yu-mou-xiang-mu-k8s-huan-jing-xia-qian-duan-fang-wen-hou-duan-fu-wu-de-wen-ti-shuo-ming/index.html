关于某项目k8s环境下前端访问后端服务的问题说明

问题描述

在甲方项目环境中，我们部署了前端服务，k8s部署文件如下：


apiVersion: apps/v1
kind: Deployment
metadata:
  name: test-batch-frontend
  namespace: test-status
spec:
  minReadySeconds: 10
  revisionHistoryLimit: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0
      maxSurge: 1
  replicas: 1
  selector:
    matchLabels:
      app: test-batch-frontend
  template:
    metadata:
      labels:
        app: test-batch-frontend
    spec:
      containers:
        - name: test-batch-frontend
          image: 192.168.223.193/test-batch/test-frontend-base-release:23
          imagePullPolicy: Always
          resources:
            requests:
              memory: 1024Mi
            limits:
              memory: 1.5Gi
          ports:
            - containerPort: 80
          env:
            - name: GATEWAY_HOST
              value: test-status-gateway.test-status.svc.cluster.local:55501

我们将后端Spring Cloud Gateway组件的地址，以k8s中的svc的完整url信息传入，但是在前端服务启动后，显示连接gateway的地址404。nginx日志如下：

后向接口明显访问不通，这样从前端页面访问，也就无法访问通后端的gateway地址，所有后端服务接口都不可达。

解决方式

1. 临时解决方式

临时解决方式很简单，修改前端服务容器nginx中的配置即可，如下：


#user  nobody;
worker_processes  4;
worker_rlimit_nofile 65535;

#error_log  logs/error.log;
#error_log  logs/error.log  notice;
#error_log  logs/error.log  info;

#pid        logs/nginx.pid;

# 环境变量
env GATEWAY_HOST;
# 加载模块信息
load_module modules/ngx_http_perl_module.so;

events {
    use epoll;
    worker_connections  204800;
}


http {
    include       mime.types;
    default_type  application/octet-stream;

    #log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
    #                  '$status $body_bytes_sent "$http_referer" '
    #                  '"$http_user_agent" "$http_x_forwarded_for"';

    #access_log  logs/access.log  main;

    sendfile        on;
    #tcp_nopush     on;

    #keepalive_timeout  0;
    keepalive_timeout  65;

    #gzip  on;
    gzip on;
    gzip_min_length  1000;
    gzip_buffers     4 8k;   
    gzip_http_version 1.1; 
    gzip_types       text/plain application/x-javascript text/css application/xml application/javascript application/json;

    client_header_buffer_size 4k; 


    #运用perl获取环境变量赋值
    perl_set $GATEWAY_HOST 'sub { return $ENV{"GATEWAY_HOST"}; }';

    server {
        listen       80;
        server_name  localhost;
        client_body_buffer_size 50m;
        client_max_body_size 100m;

        #charset koi8-r;

        #access_log  logs/host.access.log  main;

        # 前端配置
        location / {
           # 避免浏览器缓存
           if ($request_filename ~* .*\.(?:htm|html)$)
           {
              add_header Cache-Control no-cache;
              add_header Cache-Control private;
           }

            root /var/www;

            try_files $uri $uri/ @router;
            index  index.html index.htm;
            #nginx对于静态文件的处理模块，开启后会寻找以.gz结尾的文件，直接返回，不会占用cpu进行压缩。如果找不到则不进行压缩。
            gzip_static on;
        }

        location @router {
            rewrite ^.*$ /index.html last;
        }

        # 后端接口配置
        location /api/ {
            proxy_set_header Host $http_host;
            # 2. 添加完整url地址
            proxy_pass http://test-status-gateway.test-status.svc.cluster.local:55501;
            # 1. 注释掉之前的配置信息
            # proxy_pass http://$GATEWAY_HOST;
            # base项目前端适配，去掉请求中的/api/信息
            rewrite "^/api/(.*)$" /$1 break;
        }

    }

}

在配置文件中，不经过环境变量的传递了，直接写入到nginx.conf配置为文件中，重启nginx，在容器中执行nginx -s reload即可！

2. 永久解决方式

我们现有的镜像构建体系，是由基础镜像产出业务镜像，而且nginx配置信息写在基础镜像中，所以修改基础镜像中nginx如下：


#user  nobody;
worker_processes  4;
worker_rlimit_nofile 65535;

#error_log  logs/error.log;
#error_log  logs/error.log  notice;
#error_log  logs/error.log  info;

#pid        logs/nginx.pid;

# 环境变量
env GATEWAY_HOST;
# 加载模块信息
load_module modules/ngx_http_perl_module.so;

events {
    use epoll;
    worker_connections  204800;
}


http {
    include       mime.types;
    default_type  application/octet-stream;

    #log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
    #                  '$status $body_bytes_sent "$http_referer" '
    #                  '"$http_user_agent" "$http_x_forwarded_for"';

    #access_log  logs/access.log  main;

    sendfile        on;
    #tcp_nopush     on;

    #keepalive_timeout  0;
    keepalive_timeout  65;

    #gzip  on;
    gzip on;
    gzip_min_length  1000;
    gzip_buffers     4 8k;   
    gzip_http_version 1.1; 
    gzip_types       text/plain application/x-javascript text/css application/xml application/javascript application/json;

    client_header_buffer_size 4k; 


    #运用perl获取环境变量赋值
    perl_set $GATEWAY_HOST 'sub { return $ENV{"GATEWAY_HOST"}; }';

    server {
        listen       80;
        server_name  localhost;
        client_body_buffer_size 50m;
        client_max_body_size 100m;

        #charset koi8-r;

        #access_log  logs/host.access.log  main;
        # 2. k8s解析使用，GATEWAY_HOST定义为k8s内部地址，需要添加下面的dns配置，利用k8s内的dns解析解决！
        resolver kube-dns.kube-system.svc.cluster.local valid=5s;

        # 前端配置
        location / {
           # 避免浏览器缓存
           if ($request_filename ~* .*\.(?:htm|html)$)
           {
              add_header Cache-Control no-cache;
              add_header Cache-Control private;
           }

            root /var/www;

            try_files $uri $uri/ @router;
            index  index.html index.htm;
            #nginx对于静态文件的处理模块，开启后会寻找以.gz结尾的文件，直接返回，不会占用cpu进行压缩。如果找不到则不进行压缩。
            gzip_static on;
        }

        location @router {
            rewrite ^.*$ /index.html last;
        }

        # 后端接口配置
        location /api/ {
            proxy_set_header Host $http_host;
            #proxy_pass http://test-status-gateway.test-status.svc.cluster.local:55501;
            # 1. 传入信息不变
            proxy_pass http://$GATEWAY_HOST;
            # base项目前端适配，去掉请求中的/api/信息
            rewrite "^/api/(.*)$" /$1 break;
        }

    }

}

在server下添加k8s的dns解析信息，这样传入的地址，才能正常生效。后续访问就不再有问题了。

3. 存在问题

  • 不加dns解析的时候，传入相同的地址不能解析，这个想不明白是因为什么。尝试对连接进行urlencode，无效！
  • 不明白是不是地址中有特殊字符（比如 : .）导致的，尝试用双引号引入，也是无效。但是在docker中配置域名也没有问题，配置ip加端口号的方式也没有问题，在k8s中就不行。
  • 原来想在nginx.conf配置文件中添加if判断条件，判断是k8s使用的还是docker使用的，看了一篇文章后放弃了，因为ifelse在nginx配置中不可控。
