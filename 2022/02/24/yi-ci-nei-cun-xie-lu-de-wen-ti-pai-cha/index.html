一次JVM内存泄露的问题排查

问题描述

在甲方的测试环境中，莫名其妙的出现了服务掉线的情况，具体表现为服务突然从nacos下线了，有时能重新上线，有时不能。另外伴随内存突然升高，服务卡顿的现象。目前的测试收到了比较大的影响，因此进行了相应的排查操作。

问题排查

0. Arthas监控

使用Arthas进行服务监控，利用dashboard打印出当前服务的状态。但是服务的镜像是Exited状态，无法使用Arthas查看。

1. jmx监控服务运行状态

对docker运行的容器内添加jmx监控，利用prometheus+grafana的组合进行监控，找到服务掉线时的参数指标状态！

关于prometheus和grafana的具体搭建不提，这里只说明针对单个服务如何添加jmx监控项。

首先下面开始安装jmx工具，访问jmx_exporter，下载最新版本的jar包当前最新版本为：0.16.1。

然后添加jmx的配置文件如下：


startDelaySeconds: 0
ssl: false
lowercaseOutputName: false
lowercaseOutputLabelNames: false

文件命名为jmx_config.yaml，同下载的jar包放在基础镜像BaseDockerfile下的同级jmx目录中。

其次重做基础镜像，添加一个COPY层，如下：


FROM ......

WORKDIR /app
......

# 7. 添加jmx监控设施
COPY ./jmx /app/jmx

ENTRYPOINT ......

在服务所在的Dockerfile中，服务启动项命令内，添加jmx的监控信息，具体的启动项如下：


java ${JAVA_OPTS} -javaagent:/app/jmx/jmx_prometheus_javaagent-0.16.1.jar=8088:/app/jmx/jmx_config.yaml ${BUILD_PARAMS} -jar test.jar"

最后需要对暴露的8088端口进行映射，在docker启动命令里面进行映射，使用30088端口进行访问。直接访问获取的数据如下：

在面板中观察到服务异常下线的时候，发现了这样的现象

gc突然上升，而且伴随着服务下线，导致了监控数据有断层的情况，说明此时内存升高，导致资源不够用。但是服务日志并未记录异常下线的情况，通过kibana观察得不到有效的错误日志信息！

2. 服务添加内存泄露启动项

既然从服务观测上得不到有用信息，那就只能从服务本身下手了，在服务启动时，添加jvm中的内存dump选项，如下：


// 添加-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/app/heap_mem.hprof启动项
java ${JAVA_OPTS} -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/app/heap_mem.hprof -javaagent:/app/jmx/jmx_prometheus_javaagent-0.16.1.jar=8088:/app/jmx/jmx_config.yaml ${BUILD_PARAMS} -jar test.jar"

开启HeapDumpOnOutOfMemoryError这个选项，这个只会生效一次，也就是只会生成一个dump，不会把磁盘爆掉。如果开启FullGC相关选项，例如HeapDumpOnOutOfMemoryError、HeapDumpBeforeFullGC、HeapDumpAfterFullGC，如果服务在持续FullGC，将会持续产生dump文件，直到磁盘被塞满挂掉，并不能进行覆盖操作！

使用*-XX:HeapDumpPath=/app/heap_mem.hprof指定生成dump文件的地址信息，在一次服务掉线后，生成了该文件，高达2.92GB。拷贝该文件出来，导入MAT中分析。

3. MAT文件分析

如果是下载最新版的MAT工具，支持的jdk版本最低为11，无法使用jdk8。因此需要安装jdk11。

可以从这个地址找到jdk11的压缩包文件，直接解压即可，不需要再设置环境变量。

下载MAT工具，从这个地址下载，下载后的压缩包直接解压即可。

配置MAT的配置文件，编辑MemoryAnalyzer.ini，配置信息修改如下：


-vm
D:\Dev\Java\jdk-11.0.2\bin\javaw.exe
-startup
plugins/org.eclipse.equinox.launcher_1.6.200.v20210416-2027.jar
--launcher.library
plugins/org.eclipse.equinox.launcher.win32.win32.x86_64_1.2.200.v20210429-1609
-vmargs
-Xmx8192m

首先需要配置*-vm，配置jdk中的javaw.exe文件路径，这样不会影响机器中原来配置的jdk！

其次是需要把内存放大，因为heap文件太大的问题，需要大内存支撑分析。

设置完成后，保存配置信息，启动MAT工具，File->Open Heap Dump… ，找到对应路径下的heap文件，开始分析即可。时间比较长，可以先喝一杯水等等看看！

看到所有的大对象的分布情况，找到第一个Problem Suspect 1，点击See stacktrace，可以看到整个的调用栈的情况。查看所有的调用栈信息，找到对应自身业务代码的情况，如下：

之所以打码是因为有敏感信息，打码的位置是对应于业务代码内容。根据分析得知，是在数据库查询时，有一个巨大的对象留在了jvm中无法被释放，导致服务宕机的问题。

找到了调用栈，反推代码中的调用信息，在Mapper类中看到下面的代码：


......

 List<TestTaskVO> selectTestTaskVOList(@Param("testTaskVo")TestTaskVO testTaskVo,  @Param("index")Integer index, @Param("num")Integer num,@Param("plList")List<String> plList,@Param("mCList")List<String> mCList);

......

我们整个后端使用MyBatisPlus框架进行数据库查询操作，所有的查询尽可能的用分页查询的方式进行，在这个类中的查询根本就没有分页操作，这样导致了查询大数据量服务宕机的问题。

解决方案

将上述代码添加分页接口，修改如下：


IPage<TestTaskVO> selectTestTaskVOList(Page<ArVO> pageList, @Param("testTaskVo")TestTaskVO testTaskVo,  @Param("index")Integer index, @Param("num")Integer num,@Param("plList")List<String> plList,@Param("mCList")List<String> mCList);

这样查询时，返回数据量有限，就不会出现大对象消耗完jvm内存的问题了。

总结

善用工具，才能擅找问题！
